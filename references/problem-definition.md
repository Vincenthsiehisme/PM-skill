# 問題定義流程

當 PM 有模糊的訊號、用戶反饋、或觀察到異常，但還不確定根本問題是什麼時使用。

目標：從混沌的訊號，走到一個清楚的 HMW (How Might We) statement。

---

## 流程總覽

```
訊號收集 → 訊號分類 → 根本原因探索 → 問題陳述 → HMW 收斂
```

---

## Step 1：訊號收集

> 問：「你觀察到什麼讓你覺得這裡有問題？把你知道的都列出來，不用整理，先傾倒就好。」

讓 PM 自由描述：用戶說了什麼、數據顯示什麼、自己觀察到什麼。

**Claude 的工作**：
- 聆聽，不打斷
- 在 PM 說完後，把訊號整理成列表：
  ```
  訊號清單：
  - [訊號 1]（來源：[訪談/數據/觀察]）
  - [訊號 2]（來源：...）
  ```
- 標記哪些是「用戶的話」vs 「你的推論」，這兩者要分開

**完成門檻**：至少有 3 條訊號，每條都有來源。

---

## Step 2：訊號分類

將訊號分成三類：

| 類型 | 說明 | 例子 |
|------|------|------|
| **行為訊號** | 用戶做了什麼 | 「在結帳頁離開」「反覆重試」 |
| **態度訊號** | 用戶說了什麼 | 「太複雜了」「找不到在哪」 |
| **數據訊號** | 指標顯示什麼 | 「完成率 30%」「客訴量上升」 |

問 PM：「這些訊號裡，哪幾條你覺得是最重要的症狀？」

---

## Step 3：根本原因探索（5 Whys 精簡版）

針對 PM 選出的 1-2 條核心訊號，進行根本原因探索。

**節奏**：Claude 問「為什麼」，PM 回答，Claude 再問一層，直到到達「可以採取行動的根本層」。

範例對話：
```
PM：「用戶在完成設定流程前就離開了」
Claude：「他們為什麼離開？你有什麼線索嗎？」
PM：「可能覺得步驟太多」
Claude：「是步驟數量真的多，還是用戶感覺很多？有沒有觀察到他們在哪一步開始放慢？」
```

**停止條件**：
- 到了「技術限制」「法規要求」等外部約束（記錄但不繼續往下問）
- 到了「用戶的根本需求或目標」（問題的根源）
- 問了 4-5 層還沒收斂 → 暫停，標記「[需要更多用戶資料才能確認]」

**完成門檻**：能說出「根本原因假設」，格式：
```
根本原因假設：用戶 [在什麼情況下] 因為 [原因]，導致 [症狀]
```

---

## Step 4：問題陳述

根據根本原因假設，寫出問題陳述。

格式：
```
[目標用戶] 在 [情境] 需要 [達成什麼]，
但現在 [遭遇什麼阻礙]，
造成 [影響]。
```

例：
```
「第一次使用的消費者」在「完成帳戶設定」時需要「快速理解下一步」，
但現在「不知道目前進度和還剩幾步」，
造成「中途放棄，設定完成率低」。
```

> 把草稿給 PM 確認：「這樣描述你觀察到的問題，正確嗎？還是有什麼地方偏了？」

**完成門檻**：PM 確認問題陳述「沒有把解法放進去」，是純粹描述問題。

---

## Step 5：HMW 收斂

HMW（How Might We）是問題陳述的行動化轉換，目的是打開設計空間但不指定解法。

**從問題陳述轉換 HMW 的方法**：

1. **直接轉換**：把問題陳述轉成「我們如何幫助 [用戶] 在 [情境] 達成 [目標]？」
2. **移除阻礙**：「我們如何消除 [阻礙]？」
3. **改變觀點**：如果把這個「問題」當成「資源」？（通常能產生非預期的解法方向）

**輸出 3 個 HMW 版本**，讓 PM 選或調整：

```
HMW-1（移除阻礙）：我們如何讓用戶在設定流程中隨時知道進度和下一步？
HMW-2（更廣）：我們如何讓第一次使用的用戶在設定完成前不感到迷失？
HMW-3（換角度）：我們如何讓設定流程本身成為用戶想完成的事？
```

> 問 PM：「哪個版本最接近你想解決的方向？還是要調整？」

**HMW 完成門檻**：
- ✅ 有明確的目標用戶
- ✅ 有具體的情境或場景
- ✅ 沒有預設解法（不能有「透過 X 功能」「使用 Y 方式」）
- ✅ 不是「HMW 讓所有事情更好」這種萬用廢話

---

## 輸出：Problem Definition Canvas

```markdown
## Problem Definition Canvas

### 核心訊號
[列出 3-5 條最關鍵訊號，標注來源]

### 根本原因假設
[一句話：用戶在 X 情況下因為 Y 導致 Z]

### 問題陳述
[用戶 + 情境 + 需要 + 阻礙 + 影響]

### 選定的 HMW
[最終版 HMW statement]

### 尚待確認的假設
[列出 1-3 個最大的不確定性，和如何驗證]
```

---

## 從 Canvas 到 Brief

Problem Definition Canvas 完成後，詢問 PM：
> 「問題現在清楚多了。你想先評估這個機會的大小（做 Opportunity Sizing），還是直接整理成 Opportunity Brief 準備開始規劃？」
