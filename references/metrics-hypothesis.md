# 指標假設聲明（Metrics Hypothesis）

在定義分子分母之前，先把「為什麼量這個」說清楚。
**這份文件的作用是讓指標從格式正確變成思路正確。**

載入時機：定義 Metrics 任務開始時，在 metrics-guide.md 之前讀取。

## 前置步驟：讀取 Anchor

載入此文件後，先執行：
1. 搜尋對話中版本號最高的 `## ✅ Anchor vN`
2. 對照以下欄位，跳過已有答案的問診問題：

| Anchor 欄位有值 | 跳過的問診 |
|---------------|-----------|
| 本季 OKR | 問題 1（OKR 連結）完全跳過 |
| 主假設 + 假設狀態 | 問題 2（假設聲明）完全跳過 |
| 護欄指標 | 問題 3（護欄選擇邏輯）完全跳過 |
| 目標使用者 | 問題 3 的犧牲模式對照可直接套用，不需重新確認使用者類型 |

3. 若三個問題都有答案，直接進入輸出格式，產生指標假設聲明
4. 完成後更新 Anchor（加入主指標、主假設、假設狀態、失敗訊號、護欄指標）

---

## 為什麼需要指標假設聲明

一個 PM 可以把分子、分母、時間窗口全部填完，仍然定義出一個跟商業結果無關的指標。

格式正確 ≠ 指標正確。

指標假設聲明解決三個問題：
1. **OKR 連結**：這個指標對公司現在最在乎的目標有什麼貢獻
2. **假設透明**：「量這個指標會讓商業結果變好」這個連結是已驗證的，還是我們相信但尚未確認的
3. **護欄選擇邏輯**：主指標最容易被犧牲什麼，為什麼選這個護欄而不是別的

---

## 問診流程

在使用者提出要定義指標時，依序問以下問題。
**每個問題都要有實質回答，不能跳過或用一句話帶過。**

### 問題 1：OKR 連結

> 「這個功能的指標，對公司本季最重要的目標有什麼具體貢獻？
>  請說明是哪個 OKR，以及這個指標怎麼推動那個 OKR。」

**判斷回答是否有效：**

✅ 有效回答：「本季 OKR 是提升付費轉換率到 8%。這個功能讓用戶更快完成首次核心行為，我們相信縮短 Time to Value 會提升 trial-to-paid 轉換。」

❌ 無效回答：「提升用戶體驗」「增加用戶黏性」「這個功能很重要」

— 無效回答時的處理：
「你說的這個目標還沒有連結到具體的商業數字。能說明如果這個指標改善了，哪個 OKR 的數字會跟著動嗎？」

不要替使用者填這個答案，這是 PM 自己必須想清楚的事。

---

### 問題 2：假設聲明

> 「你相信這個指標和商業結果之間有什麼連結？
>  這個連結是已有數據支持，還是邏輯推演？」

**目標格式（引導使用者產出）：**

```
我們相信：[使用者行為改變] 會導致 [中間結果] 進而影響 [商業結果]

這個連結目前是：
  □ 已有歷史數據支持（說明數據來源）
  □ 邏輯推演，尚未驗證
  □ 這次上線本身就是要驗證這個假設

如果假設錯了，我們會看到：[什麼訊號告訴我們這條鏈是斷的]
```

**範例：**
```
我們相信：用戶在 Day-3 完成個人化設定，會建立使用習慣，進而提升 D30 留存率

這個連結目前是：
  ✓ 已有歷史數據支持 — 過去 cohort 分析顯示完成設定的用戶 D30 留存率高出未完成者 23%

如果假設錯了，我們會看到：設定完成率上升，但 D30 留存率沒有跟著動
```

**重要限制：**
- 不要幫使用者編造這條鏈，如果他說不清楚，讓他說「邏輯推演，尚未驗證」
- 「如果假設錯了」這一行必須是可觀察的訊號，不能是「一切都沒有改變」這種廢話

---

### 問題 3：護欄選擇邏輯

> 「如果這個主指標照計畫改善了，最容易在哪裡出現副作用？」

不是問「有哪些護欄指標」，而是問「為了讓主指標好看，最容易犧牲什麼」。

**常見的犧牲模式（根據主指標類型）：**

| 主指標類型 | 最容易被犧牲的 | 對應護欄 |
|-----------|--------------|---------|
| 採用率 ↑ | 用戶完成任務的時間變長（強迫用戶使用） | 任務完成時間 ≤ 現況 |
| 轉換率 ↑ | 用戶品質下降（吸引到不對的用戶） | D30 留存率 ≥ 現況 |
| 功能使用頻率 ↑ | 頁面整體 engagement 下降（功能搶了其他功能的注意力） | 整體 session 長度 ≥ 現況 |
| 完成率 ↑ | 客訴量上升（強迫用戶完成他們不想完成的流程） | 客訴率 ≤ 現況 |
| 點擊率 ↑ | 後續流程 drop-off 上升（點了但沒有真的要用） | 點擊後完成率 ≥ 現況 |

引導使用者選護欄時，要求他說明「為什麼選這個，而不是其他可能的副作用」。

## 輸出格式

問診完成後，輸出一份指標假設聲明，附在 metrics-guide.md 的指標定義之前：

```markdown
### 指標假設聲明

**OKR 連結**
本季 OKR：[OKR 名稱和目標數字]
貢獻方式：[這個指標怎麼推動那個 OKR，一到兩句話]

**假設聲明**
我們相信：[行為改變] → [中間結果] → [商業結果]
連結狀態：[已驗證 / 邏輯推演 / 本次上線驗證]
驗證來源：[數據來源，若已驗證]
失敗訊號：[如果假設錯了，會看到什麼]

**護欄選擇理由**
主指標：[指標名稱]
最可能的副作用：[描述]
對應護欄：[護欄指標名稱] [≥/≤] [門檻值]
選擇理由：[為什麼這個副作用比其他副作用更值得監控]
```

## 完成標準

指標假設聲明算「完成」需滿足：

- [ ] OKR 連結有具體的 OKR 名稱和數字，不是泛稱「提升用戶體驗」
- [ ] 假設聲明有明確的行為改變 → 中間結果 → 商業結果三層
- [ ] 連結狀態已標明（已驗證 / 邏輯推演 / 本次上線驗證）
- [ ] 失敗訊號是可觀察的具體訊號，不是「什麼都沒改變」
- [ ] 護欄選擇有明確的「為什麼選這個副作用」的理由

---

## Gotchas

- **OKR 連結不能由 Claude 替代**：如果使用者說不清楚跟 OKR 的連結，不要幫他填，讓他說「本季無明確 OKR 對應，這個功能是技術債 / 合規要求 / 探索性」。誠實的標記比虛假的連結有價值
- **假設聲明不是因果鏈**：不要要求使用者寫出完整的五層因果鏈，三層（行為 → 中間結果 → 商業結果）已經足夠，更多層會變成說故事而不是思考
- **「邏輯推演」是合法的狀態**：大多數功能上線時，假設都是未驗證的。承認這一點比假裝有數據支持更誠實，也更有用
- **護欄不是越多越好**：一個主指標對應一到兩個護欄，不是越多越安全。護欄太多代表 PM 不知道最需要保護什麼
- **失敗訊號必須在上線後第幾週可以觀察到**：「D30 留存沒有動」代表要等 30 天才知道假設對不對，這太慢了。好的失敗訊號是「上線後第二週 Day-7 留存沒有改善」
