# 狀態快照（Status Snapshot）

每次輸出任何章節或任務內容後，自動在回應末尾附上目前狀態。
不宣告「完成」，不給導覽選單。只提供透明度，讓 PM 自己判斷要繼續哪個部分。

## 前置步驟：讀取 Anchor

若對話中存在 Anchor Statement：
- 直接從 Anchor 的「PRD 章節狀態」欄位讀取各章節狀態，不需重新掃描對話
- 輸出快照後，確認 Anchor 的章節狀態欄位與快照一致，若不一致以快照為準並更新 Anchor

---

## 格式

```
─────────────────────────────
📋 目前狀態
背景      ✅  /  目標      ⚠️  /  使用者    ✅
功能需求  ❌  /  技術考量  ─   /  開放問題  ⚠️

⚠️ 需要注意：[最關鍵的 1 個缺口，一句話說明]
─────────────────────────────
```

## 符號定義

| 符號 | 意思 |
|------|------|
| ✅ | 達到品質門檻 |
| ⚠️ | 有內容但未達門檻（空洞或缺要素） |
| ❌ | 空白或缺失 |
| ─ | 本次未涉及，不評分 |

## 規則

- **只顯示已討論過的章節**：沒有討論過的章節用 `─`，不預先顯示所有欄位
- **「需要注意」只挑一個**：選最影響後續進展的缺口，不列清單，不給優先級排序
- **不主導節奏**：不說「接下來建議你做 X」，讓 PM 自己決定下一步
- **可以關閉**：若 PM 說「不用顯示狀態」或「繼續就好」，停止附加快照，不再提醒

## 品質門檻對照（判斷 ✅/⚠️/❌ 的依據）

| 章節 | ✅ 達門檻 | ⚠️ 未達門檻 | ❌ |
|------|---------|------------|-----|
| 背景 | 誰受影響 + 規模 + Why now | 缺任一項 | 空白 |
| 目標 | 有分子分母的可量測指標 | 只有文字目標 | 空白 |
| 使用者 | 角色 + 情境 + workaround | 缺任一項 | 空白或「一般用戶」 |
| 功能需求 | P0 ≥1 則且有 AC | 有 Story 缺 AC | 空白或只有功能列表 |
| 技術考量 | 有內容或明確 [待工程確認] | 模糊描述 | 空白 |
| UI/UX | 有流程描述或設計稿連結 | 只寫「待設計」 | 空白 |
| 開放問題 | 有清單或明確「無開放問題」 | — | 空白 |

## Gotchas

- **不要在第一次輸出前就顯示快照**：快照是輸出之後的附件，不是輸出之前的預告
- **⚠️ 需要注意只能一條**：多條等於沒有優先級，使用者不知道從哪裡開始
- **User Story 和 Metrics 的狀態不列在快照裡**：快照只追蹤 PRD 章節；Story 和 Metrics 的完成狀態由各自的完成標準處理
