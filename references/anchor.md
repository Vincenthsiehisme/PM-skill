# Anchor Statement

Anchor 是整個工作流程的共享 context 文件。
每個階段完成後更新它，下一個階段從它開始，跳過已有答案的問診。

---

## 讀取規則（所有 reference 文件都必須遵守）

1. 在執行任何問診之前，先搜尋對話中**版本號最高**的 Anchor Statement
2. 若找到 Anchor，對照欄位，跳過已有答案的問診問題
3. 若某欄位是 `[待確認]`，該問診問題仍然需要問
4. 若對話中沒有任何 Anchor，從空白開始正常問診

---

## 更新規則

- 每個階段完成後，輸出更新後的 Anchor（版本號 +1）
- 新版本必須包含舊版本的所有欄位，不能遺漏
- 加上「取代 vN」的聲明，讓 Claude 知道哪個是最新版本
- 有改動的欄位用 `←更新` 標記，方便辨識

---

## Anchor 模板

```markdown
## ✅ Anchor v[N]（取代 v[N-1]）
更新於：[階段名稱]（PRD 背景 / Metrics 問診 / Story 拆解 / 其他）

### 產品基本資訊
功能名稱：
目標使用者：（角色 + 使用情境 + 現有 workaround）
本季 OKR：（名稱 + 目標數字）
功能範圍摘要：（一句話說明 P0 的核心是什麼）

### 關鍵決策
主指標：（名稱 + 類型）
主假設：（行為 → 中間結果 → 商業結果）
假設狀態：（已驗證 / 邏輯推演 / 本次上線驗證）
失敗訊號：（如果假設錯了，會看到什麼）
護欄指標：（名稱 + 門檻）

### PRD 章節狀態
背景：（✅ / ⚠️ / ❌ / ─）
目標與指標：（✅ / ⚠️ / ❌ / ─）
目標使用者：（✅ / ⚠️ / ❌ / ─）
功能需求：（✅ / ⚠️ / ❌ / ─）
技術考量：（✅ / ⚠️ / ❌ / ─）
UI/UX：（✅ / ⚠️ / ❌ / ─）
開放問題：（✅ / ⚠️ / ❌ / ─）

### 待確認項目
- [ ] [項目]（來源章節）

### 改動記錄
- v1：初始化（[階段]）
- v[N]：[改動摘要]
```

---

## 各階段對應的更新欄位

| 完成的階段 | 應更新的欄位 |
|-----------|------------|
| PRD 背景問診 | 功能名稱、目標使用者、本季 OKR、PRD 章節狀態（背景） |
| PRD 目標問診 | 功能範圍摘要、PRD 章節狀態（目標與指標） |
| Metrics 問診 | 主指標、主假設、假設狀態、失敗訊號、護欄指標 |
| User Story 拆解 | 功能範圍摘要（若有調整）、PRD 章節狀態（功能需求） |
| 版本迭代 | 任何改動的欄位 + 改動記錄 |

---

## Anchor 欄位對各 reference 的影響

| Anchor 欄位 | 讓哪個 reference 跳過哪個問診 |
|------------|---------------------------|
| 目標使用者 | prd-template Section 3 全部問診；user-story-guide 的 who 不需再問 |
| 本季 OKR | metrics-hypothesis 問題 1（OKR 連結）跳過 |
| 主假設 + 假設狀態 | metrics-hypothesis 問題 2（假設聲明）跳過 |
| 護欄指標 | metrics-hypothesis 問題 3（護欄選擇）跳過 |
| 功能範圍摘要 | user-story-guide 的功能描述不需再問；roadmap-guide 的項目清單可直接帶入 |
| PRD 章節狀態 | status-snapshot 直接讀欄位，不需重新掃描對話 |

## Gotchas

- **版本號是唯一的真相機制**：若對話中有多個 Anchor，永遠以版本號最高的為準，不管它在對話的哪個位置
- **欄位空白 ≠ 已確認為空**：若某欄位完全沒有出現，代表尚未問診。若欄位是 `[待確認]`，代表已問但答案未定
- **不要因為 Anchor 有值就跳過所有確認**：Anchor 記錄的是「已確認的決策」，但使用者可能在後續對話中改變心意。若使用者明確說「我改一下」，優先以新說法為準，然後更新 Anchor
- **改動記錄要具體**：「v3：更新目標使用者」不夠，要寫「v3：目標使用者從新註冊用戶改為付費用戶，影響 Metrics 分母定義」
- **Anchor 不是 PRD**：它是 context 的摘要，不是完整文件。欄位要精簡，不要把整個章節內容貼進來
