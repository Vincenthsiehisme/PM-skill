# PRD Template & 問診流程

PRD 的每個章節都有對應的「問診問題」。
**在填寫每個章節之前，先確認使用者能回答這些問題。**
回答不出來的問題，在對應欄位標記 [待確認] 並列入開放問題清單。

## 前置步驟：讀取 Anchor

載入此文件後，先執行：
1. 搜尋對話中版本號最高的 `## ✅ Anchor vN`
2. 對照以下欄位，跳過已有答案的問診問題：

| Anchor 欄位有值 | 跳過的問診 |
|---------------|-----------|
| 目標使用者 | Section 3 全部問診問題 |
| 本季 OKR | Section 2 問診問題 1（這個功能成功了你怎麼知道）中的 OKR 確認 |
| 功能範圍摘要 | Section 4 問診問題 1（如果只能做一件事） |

3. 若 Anchor 不存在，正常跑完整問診，並在 Section 1 完成後初始化 Anchor v1

## 標準 PRD 結構

### 文件資訊
```
功能名稱：
文件版本：v0.1
作者：
最後更新：
狀態：草稿 / 審閱中 / 確認
```

---

### Section 1：背景與問題定義

**問診問題（寫背景前先問）：**
1. 現在沒有這個功能時，使用者怎麼解決這個問題？（釐清 workaround，代表真實痛點）
2. 這個問題影響多少人？每天都遇到，還是偶爾？（釐清規模與頻率）
3. 為什麼是現在做，而不是三個月後？（釐清 urgency，回答不出來代表優先級可能有問題）

**填寫格式：**
- **問題陳述**：[誰] 在 [什麼情境] 下遇到 [什麼問題]，目前靠 [workaround] 解決
- **Why now**：[觸發因素 — 業務目標、用戶反饋量、競品動態等]
- **不解決的代價**：[若跳過這個功能，會發生什麼具體的事]

**完成門檻：** 三項都有實質內容 = ✅；缺規模或 urgency = ⚠️；空白或一句話帶過 = ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

---

### Section 2：目標與成功指標

**問診問題：**
1. 這個功能成功了，你怎麼知道它成功了？（引出可量測指標）
2. 你現在有這個指標的 baseline 數字嗎？（沒有就要先建立量測）
3. 有沒有不希望因為這個功能而變差的指標？（引出 guardrail metrics）

**填寫格式：**
- **產品目標**：[對應業務目標的 1-2 句話]
- **成功指標**：（見 metrics-guide.md 定義格式）
- **不在範圍內**：[明確列出這次不做什麼，防止 scope creep]

**完成門檻：** 至少 1 個含分子分母的可量測指標 = ✅；只有文字目標 = ⚠️；空白 = ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

---

### Section 3：目標使用者

**問診問題：**
1. 這個功能的主要使用者是誰？他們的日常工作是什麼？
2. 他們什麼時候、在什麼裝置上會用到這個功能？
3. 現在他們怎麼解決這個問題？（與 Section 1 呼應，但從使用者視角描述）

**填寫格式：**
- **主要使用者**：[職稱/角色] + [行為特徵描述，不只是職稱]
- **使用情境**：[什麼時候] + [在哪裡] + [用什麼裝置]
- **現有 workaround**：[他們現在怎麼做，這說明功能替代方案是什麼]

**完成門檻：** 角色 + 情境 + workaround 都有 = ✅；只有角色 = ⚠️；只寫「一般用戶」= ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

---

### Section 4：功能需求

**問診問題：**
1. 如果只能做一件事，最重要的是什麼？（定出 P0）
2. 這個功能的正常流程是什麼？從使用者打開到完成，每一步是什麼？（引出 happy path）
3. 有沒有不同角色看到的東西不一樣？（引出 permission story）
4. 如果中途失敗了，使用者會看到什麼？（引出 error handling）

**填寫格式：**
用 User Story 格式列出（見 user-story-guide.md）

優先級：
- P0：MVP 必須有，沒有就不上線
- P1：重要但可以第二期
- P2：Nice to have

**完成門檻：** P0 ≥1 則且每則有 AC = ✅；有 Story 但缺 AC = ⚠️；只有功能列表 = ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

---

### Section 5：系統架構 Context

**問診問題：**
1. 這個功能涉及哪些現有的系統或服務？
2. 資料從哪裡來？需要新的資料來源嗎？
3. 這個功能跟現有哪個功能有關係或相依？

若使用者無法回答，整節標記 [待確認] 並列入開放問題，不要猜測。

**填寫格式：**
- 涉及的平台 / 服務：
- 資料來源：
- 與現有功能的相依關係：

---

### Section 6：技術考量

**問診問題：**
1. 有沒有已知的技術限制或風險？
2. 有沒有第三方服務的依賴？
3. 這個功能有沒有資安或隱私的疑慮？（資料收集、權限控制）

若無法回答，整節標記 [待工程確認]。

**填寫格式：**
- 已知技術限制：
- 第三方依賴：
- 資安 / 隱私考量：

**完成門檻：** 有實質內容或明確 [待工程確認] = ✅；空白 = ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

---

### Section 7：UI/UX 需求

**問診問題：**
1. 有設計稿或 wireframe 嗎？
2. 沒有設計稿的話，關鍵的操作流程是什麼？（用文字描述即可）

**填寫格式：**
- 關鍵畫面或流程描述：
- 設計稿連結：（如有）

**完成門檻：** 有文字流程描述或設計稿連結 = ✅；只寫「待設計」但無任何描述 = ⚠️；空白 = ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

### Section 8：開放問題

列出所有還沒有答案的問題。

格式：
| 問題 | 負責人 | 截止日 | 狀態 |
|------|--------|--------|------|

**完成門檻：** 有清單或明確「本次無開放問題」= ✅；空白 = ❌

**→ 輸出章節後停下來，等待 PM 回應 A / B / C 再繼續。**

## 完成標準

PRD 每個章節算「完成」的最低門檻。
輸出前逐章對照，未達門檻的章節加上 ⚠️ 標記，不能靜默跳過。

| 章節 | 最低門檻 |
|------|---------|
| 背景與問題 | 說明「誰受影響」、「影響規模/頻率」、「Why now」三項，缺一標記 ⚠️ |
| 目標與指標 | 至少 1 個可量測指標（有分子分母），純文字目標不算完成 |
| 目標使用者 | 說明角色、使用情境、現有 workaround，不能只寫職稱 |
| 功能需求 | P0 Story 至少 1 則，且每則都有 AC |
| 技術考量 | 有內容或明確標記 [待工程確認]，不能空白 |
| UI/UX 需求 | 有文字流程描述或設計稿連結；無設計稿時至少描述關鍵操作步驟 |
| 開放問題 | 有內容或明確標記「本次無開放問題」，不能空白 |

章節完成度計算（供 build_report 使用）：
- 完整填寫且達門檻 = ✅ 1 分
- 有內容但未達門檻 = ⚠️ 0.5 分
- 空白或只有 [待確認] = ❌ 0 分
