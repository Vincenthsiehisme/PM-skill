# Roadmap 規劃指南

## 使用時機

當使用者說「roadmap」「產品規劃」「排優先級」「Now Next Later」「季度規劃」時載入此文件。

---

## 問診問題（開始前必問）

1. Now / Next / Later 各代表幾個月？（**必問，不能假設**）
   - 常見設定：Now = 本季、Next = 下季、Later = 半年後以上
   - 若使用者說不確定，建議用「Now = 0-3個月、Next = 3-6個月、Later = 6個月以上」
2. 目前有哪些待規劃的功能或項目？（請列出清單）
3. 有沒有已確定的硬性截止日或業務承諾？（這些是 Now 的錨點）
4. 排序的主要考量是什麼？（用戶價值 / 業務收益 / 技術債 / 戰略佈局）

---

## 排序框架

### Impact vs Effort 矩陣

先讓使用者對每個項目做粗略評估：

| 項目 | Impact（用戶/業務價值） | Effort（開發成本） | 象限 |
|------|----------------------|------------------|------|
| [項目] | 高/中/低 | 高/中/低 | [見下方] |

象限判斷：
- **高 Impact + 低 Effort** → Quick Win，優先放 Now
- **高 Impact + 高 Effort** → Strategic Bet，放 Next 並拆解
- **低 Impact + 低 Effort** → Fill-in，有空才做
- **低 Impact + 高 Effort** → 重新考慮是否要做

### 硬性約束優先

在套用矩陣之前，先把這些固定在時間軸上：
- 已承諾給客戶或合作夥伴的功能
- 合規或法規截止日
- 依賴其他團隊的項目（受限於他人的 timeline）

---

## 輸出格式

```markdown
## Roadmap：[產品/功能範圍]

**規劃日期：** [日期]
**時間定義：**
- Now：[日期範圍]
- Next：[日期範圍]
- Later：[日期範圍以後]

### Now（[日期範圍]）
| 項目 | 優先理由 | 負責人 | 預計完成 |
|------|---------|--------|---------|
| [項目] | [為什麼現在] | | |

### Next（[日期範圍]）
| 項目 | 優先理由 | 前置條件 |
|------|---------|---------|
| [項目] | [為什麼下一期] | [需要什麼才能開始] |

### Later（[日期範圍以後]）
| 項目 | 備注 |
|------|------|
| [項目] | [方向性說明，不需精確] |

### 暫不規劃
| 項目 | 原因 |
|------|------|
| [項目] | [為什麼現在不做] |
```

---

## 完成標準

- [ ] Now / Next / Later 的時間範圍已明確定義（不能模糊）
- [ ] 每個 Now 項目有明確的優先理由（不只是「重要」）
- [ ] 硬性約束項目已識別並固定在對應 horizon
- [ ] Next 項目有前置條件說明
- [ ] 有「暫不規劃」清單，說明為什麼不做（這和 PRD 的 out of scope 一樣重要）

## Gotchas

- **horizon 未定義就輸出是最常見的失敗**：Now 對不同人意思完全不同，不問清楚就輸出等於廢紙
- **Roadmap 不是承諾**：輸出時在頂部加一行「本 roadmap 為規劃方向，非交付承諾，會隨業務優先級調整」
- **Later 不是垃圾桶**：Later 的項目也要有進入條件（什麼情況下會升到 Next），否則它永遠不會被執行
- **Impact 是主觀的**：若使用者和 stakeholder 對 Impact 的判斷不同，這本身就是一個開放問題，要列進去
